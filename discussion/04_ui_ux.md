# 4. UI/UX 설계

## 토론 참여자
- PM (기획자) / FE (프론트엔드 개발자) / ECO (경제 도메인 전문가) / DA (데이터 아키텍트)

---

### FE: 현재 프로토타입의 UI 문제점부터 짚겠습니다

현재 index.html을 모바일에서 열어보면:

1. **노드가 화면에 비해 작음** — 32px 반지름이 모바일에서 터치하기 힘듦 (권장 최소 44px)
2. **노드 위치가 고정 좌표 기반** — 화면 크기 비례 스케일링이지만 작은 폰에서 겹침
3. **팝업이 화면 45% 차지** — 작은 폰에서 관계도가 거의 안 보임
4. **범례가 왼쪽 상단에 항상 노출** — 공간 낭비
5. **핀치 줌/패닝 없음** — 노드가 많아지면 필수
6. **다크 테마만** — 괜찮지만, 텍스트 가독성이 좀 낮음 (특히 #999 서브텍스트)

### PM: 사용자 시나리오별 UI 흐름을 정의합시다

**시나리오 1: 탐색 (가장 빈번)**
1. 앱 열기 → 전체 관계도 보임
2. 관심 노드 탭 → 하이라이트 + 팝업
3. "연쇄 효과" 버튼 → 파급 경로 시각화
4. 다른 노드 탭하거나 빈 곳 탭 → 해제

**시나리오 2: 시뮬레이션**
1. 노드 탭 → 팝업
2. "오르면?" 또는 "내리면?" 버튼
3. 전체 관계도에 영향 방향 표시
4. 특정 노드 탭 → 해당 경로의 상세 설명

**시나리오 3: 학습 (심화)**
1. 노드 탭 → 팝업에서 상세 설명 읽기
2. 관련 지표 링크 따라가며 탐색
3. 관계선 탭 → 관계 설명 읽기

### FE: 화면 구조 제안

```
+----------------------------------+
|  [검색] [필터] [메뉴]  (상단바)   |
+----------------------------------+
|                                  |
|                                  |
|        관계도 영역 (SVG)          |
|       핀치줌/패닝 가능            |
|                                  |
|                                  |
+----------------------------------+
|  팝업 (하단 시트, 스와이프업)      |
|  - 지표명, 설명                   |
|  - [오르면?] [내리면?] 버튼       |
|  - 연결된 지표 목록               |
+----------------------------------+
```

핵심 설계 원칙:
- **하단 시트(Bottom Sheet)** 패턴: 모바일 네이티브 앱 느낌. 스와이프 위로 확장, 아래로 닫기.
- **관계도는 항상 화면 전체**: 팝업이 떠도 관계도 일부가 보여야 함 (30% 이상)
- **FAB(Floating Action Button)**: 시뮬레이션 모드 진입용 (하단 우측)

### ECO: 정보 구조에 대해 의견 있습니다

노드 팝업에 들어가야 할 정보 우선순위:

1. **지표명 + 카테고리** (항상 보임)
2. **한줄 설명** (항상 보임)
3. **"오르면?/내리면?" 버튼** (항상 보임) — 이게 핵심 CTA
4. 연결된 지표 목록 (스크롤 시 보임)
5. 해석 가이드: 이 숫자의 의미 (펼치면 보임)
6. 발표 일정 (펼치면 보임)
7. 관련 메모 (Phase 2)

중요한 건, **"오르면?/내리면?" 버튼이 가장 눈에 띄는 곳에** 있어야 한다는 겁니다. 이게 이 앱의 킬 피처니까요.

### PM: 색상 체계도 재정비가 필요합니다

현재:
- 노드 색: 카테고리별 (10가지)
- 선 색: 비례(초록)/반비례(빨강)/조건부(노랑) — 3가지

문제: 시뮬레이션 모드에서 "영향 방향"을 색으로 표시하면, 기존 색과 충돌합니다.

제안:
- **기본 모드**: 노드=카테고리 색, 선=관계 타입 색 (현재와 동일)
- **시뮬레이션 모드**: 노드에 아이콘(▲▼) 오버레이, 선은 **두께/애니메이션(점선 흐름)**으로 활성 경로 표시
- 이러면 색 충돌 없이 모드 전환 가능

### FE: 기술적 구현 관련 논의

**D3.js vs 순수 SVG:**

현재 순수 SVG로 되어 있는데, force-directed layout과 애니메이션을 넣으려면 선택이 필요합니다.

옵션 A: **D3.js** (d3-force)
- 장점: force simulation 내장, 풍부한 예제, 커뮤니티
- 단점: 러닝 커브, 번들 크기 (CDN으로 해결 가능)

옵션 B: **순수 JS + 간단한 force 알고리즘 직접 구현**
- 장점: 의존성 없음, 가벼움
- 단점: 직접 구현해야 해서 시간 소요, 엣지 케이스

옵션 C: **Cytoscape.js**
- 장점: 그래프 전용 라이브러리, 레이아웃/이벤트/스타일 내장
- 단점: D3보다 덜 유연, 비주얼 커스터마이징 제한

**추천: D3.js**
- force layout이 핵심이므로 d3-force가 가장 적합
- CDN 한 줄이면 됨
- 풍부한 레퍼런스

### DA: 모바일 성능 이슈 주의

D3 force simulation은 노드가 많아지면 매 프레임 N^2 계산을 합니다. 현재 12개면 괜찮지만, 30-50개 노드가 되면 저사양 폰에서 버벅일 수 있어요.

**대응:**
1. force simulation은 초기 배치에만 사용하고, 안정화(alpha < 0.01)되면 멈추기
2. 노드 50개 제한 (실용적으로도 50개 넘으면 가독성 문제)
3. requestAnimationFrame 기반 렌더링

### PM: 반론 — 정말 50개 넘게 갈 수 있나요?

현실적으로 경제 지표 관계도에 50개 이상의 노드가 필요한가요?

### ECO: 필요합니다, 장기적으로는

현재 12개는 최소한의 핵심 지표예요. 실제로는:
- 주요 지표: 20-25개 (고용, GDP, PMI, 소비자심리, 무역수지 등 추가)
- 섹터별 주식: 10-15개 (반도체, 바이오, 에너지, 방산 등)
- 개별 종목/ETF: 사용자별로 5-10개
- 이벤트/정책: 5-10개 (대선, 관세, 규제 등)

총 40-60개 정도가 현실적 상한이에요.

### FE: 그렇다면 **그룹핑/접기** 기능이 필요하겠네요

60개 노드를 다 펼쳐놓으면 어떤 화면에서도 혼란스럽습니다. 대안:

1. **카테고리 그룹 접기**: "주식" 카테고리 탭하면 하위 노드들이 접히거나 펼쳐짐
2. **관심 지표 중심 뷰**: 내가 관심 있는 5-10개 지표만 보여주는 포커스 모드
3. **레벨별 보기**: 기본(핵심 12개) → 확장(25개) → 전체(60개) 토글

**MVP에서는 핵심 노드만(현재 12개 + 알파), Phase 2에서 그룹핑 추가**가 현실적입니다.

## 합의된 UI/UX 설계

### 모바일 화면 구조
- 상단: 검색바 + 필터 아이콘 + 메뉴(햄버거)
- 중앙: 관계도 (SVG, 핀치줌/패닝 지원)
- 하단: 바텀시트 (노드 정보, 시뮬레이션 버튼)

### 인터랙션
- **탭**: 노드 선택 → 1차 연결 하이라이트 + 바텀시트 팝업
- **바텀시트 "오르면?/내리면?"**: 시뮬레이션 모드 진입
- **바텀시트 스와이프 다운**: 닫기
- **빈 곳 탭**: 선택 해제
- **노드 드래그**: 위치 조정 (유지)
- **핀치 줌/패닝**: 관계도 전체 확대/축소/이동

### 시각화 라이브러리
- **D3.js** (d3-force, d3-zoom, d3-drag) — CDN
- Force-directed layout 기본, 사용자 드래그 위치 저장

### 색상 체계
- 노드: 카테고리별 색상 (유지)
- 선: 비례(초록)/반비례(빨강)/조건부(노랑) (유지)
- 시뮬레이션 모드: ▲(초록)/▼(빨강)/?(회색) 아이콘 오버레이

### 성능 고려
- Force simulation: 초기 배치 후 정지
- 노드 상한: ~50개 (Phase 2에서 그룹핑으로 대응)
- 노드 터치 영역: 최소 44px

### MVP UI 범위
- 전체 관계도 + 핀치줌/패닝
- 노드 탭 → 바텀시트 (기본 정보 + 시뮬레이션 버튼)
- 카테고리 필터 (상단)
- 검색 (상단)
- 다크 테마 (유지, 가독성 개선)
