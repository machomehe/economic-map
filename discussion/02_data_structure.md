# 2. 데이터 구조 설계

## 토론 참여자
- PM (기획자) / FE (프론트엔드 개발자) / ECO (경제 도메인 전문가) / DA (데이터 아키텍트)

---

### DA: 현재 구조의 문제점부터 짚겠습니다

현재 `index.html`에 하드코딩된 데이터:

```js
// 노드: 12개, 필드: id, label, sub, cat, x, y, desc
// 링크: 15개, 필드: source, target, relation, color, desc
```

**문제점:**
1. 데이터와 UI 코드가 완전히 혼합 — 데이터만 수정하기 어려움
2. 관계의 메타데이터가 부족 — 강도, 조건, 시간 프레임 없음
3. 노드 위치(x, y)가 데이터에 포함 — 레이아웃은 뷰 레이어 관심사인데 데이터와 섞여 있음
4. 확장성 없음 — 노드/링크 추가하려면 코드 직접 수정 필요

### ECO: 경제적 관점에서 필요한 데이터 구조를 말하겠습니다

지표를 제대로 표현하려면 현재보다 훨씬 많은 정보가 필요합니다:

**노드(지표)에 필요한 정보:**
- 기본: id, 이름, 부제, 카테고리
- 핵심 설명: 이 지표가 뭔지 (초보 대상)
- 해석 가이드: 이 숫자가 올라가면/내려가면 무슨 뜻인지
- 주요 발표일: CPI는 매월 둘째 주, FOMC는 6주마다 등
- 관련 티커/데이터소스: FRED 코드, Yahoo Finance 심볼 등
- 현재값 & 추세 (동적, 향후 연동)

**링크(관계)에 필요한 정보:**
- 기본: source, target, 관계 방향(비례/반비례)
- **강도**: 강함/중간/약함 (또는 1-5 스케일)
- **시간 프레임**: 즉각적/단기(1-3개월)/중기(3-12개월)/장기(1년+)
- **조건**: 어떤 상황에서 이 관계가 성립하는지
- **반례/예외**: 이 관계가 깨지는 경우
- **근거/출처**: 왜 이 관계가 성립하는지 (학습 목적)
- **추가일/수정일**: 언제 추가되었고 최근 검증은 언제인지

### FE: 잠깐요, 너무 복잡하면 초보 사용자가 데이터를 직접 관리 못합니다

ECO 말이 이론적으로는 다 맞는데, 현실적으로 생각해야 해요. 사용자가 JSON 파일을 직접 편집해야 할 수 있는데, 필드가 20개면 실수가 날 거예요.

**제안: 계층적 구조 (필수 + 선택)**
- Level 1 (필수): id, label, category, description — 이것만 있으면 동작
- Level 2 (권장): 관계 강도, 시간 프레임
- Level 3 (고급): 조건, 반례, 출처 등

이렇게 하면 처음에는 간단하게 시작하고, 점점 풍부하게 만들 수 있어요.

### DA: 좋은 접근입니다. 구체적 JSON 스키마를 제안하겠습니다

```
data/
  nodes.json      — 지표 목록
  links.json      — 관계 목록
  categories.json — 카테고리 정의 (색상 등)
  events.json     — 이벤트/시나리오 (향후)
```

**nodes.json 스키마:**
```json
{
  "version": "1.0",
  "lastUpdated": "2026-04-03",
  "nodes": [
    {
      "id": "rate",
      "label": "기준금리",
      "sub": "Fed Funds Rate",
      "category": "금리",
      "description": "연준이 설정하는 기준금리...",
      "interpretation": {
        "up": "긴축 신호. 대출 비용 증가, 투자 위축",
        "down": "완화 신호. 대출 비용 감소, 투자 촉진"
      },
      "schedule": "FOMC 연 8회 (약 6주 간격)",
      "dataSource": {
        "fred": "FEDFUNDS",
        "frequency": "daily"
      },
      "tags": ["핵심", "선행지표"]
    }
  ]
}
```

**links.json 스키마:**
```json
{
  "version": "1.0",
  "lastUpdated": "2026-04-03",
  "links": [
    {
      "id": "rate-bond",
      "source": "rate",
      "target": "bond",
      "type": "반비례",
      "strength": 5,
      "timeframe": "즉각",
      "description": "금리↑ → 채권가격↓",
      "detail": "금리와 채권가격은 수학적으로 역관계...",
      "conditions": "일반적으로 항상 성립 (가장 확실한 관계)",
      "exceptions": "신용 위기 시 국채는 안전자산으로 가격 상승 가능",
      "source_note": "기본 채권 이론",
      "addedAt": "2026-04-03",
      "addedBy": "initial"
    }
  ]
}
```

### PM: 이벤트/시나리오 데이터도 필요할 것 같은데요?

사용자의 핵심 유스케이스가 "경제 톡방에서 배운 걸 기록"이라면, **이벤트 기반 시나리오**를 저장하는 구조가 필요합니다:

예: "2026년 3월 FOMC — 금리 동결 결정"
→ 이 이벤트가 어떤 지표에 어떤 영향을 미쳤는지 기록
→ 시간이 지나면 "이때 실제로 어떻게 됐는지" 검증

### DA: 좋은 포인트입니다. events.json을 추가하죠

```json
{
  "events": [
    {
      "id": "fomc-2026-03",
      "title": "2026년 3월 FOMC 금리 동결",
      "date": "2026-03-19",
      "type": "정책",
      "affectedNodes": ["rate", "bond", "dollar"],
      "impacts": [
        {
          "node": "rate",
          "direction": "유지",
          "note": "시장 예상대로 동결. 6월 인하 기대 유지"
        }
      ],
      "memo": "톡방에서 '금리 동결이지만 점도표가 비둘기파'라는 분석",
      "source": "경제톡방",
      "addedAt": "2026-03-20"
    }
  ]
}
```

### ECO: 이벤트 구조에 추가로 **"시나리오 체인"** 개념이 필요합니다

단일 이벤트의 영향뿐 아니라, **시나리오 경로**를 저장할 수 있어야 해요:

```json
{
  "scenarios": [
    {
      "id": "war-escalation",
      "title": "중동 전쟁 확대 시나리오",
      "chain": [
        {"node": "war", "direction": "↑", "order": 1},
        {"node": "oil", "direction": "↑", "order": 2},
        {"node": "inflation", "direction": "↑", "order": 3},
        {"node": "rate", "direction": "↑(압력)", "order": 4},
        {"node": "growth", "direction": "↓", "order": 5}
      ],
      "probability": "중간",
      "myAction": "성장주 비중 축소, 에너지주/금 비중 확대 고려"
    }
  ]
}
```

이렇게 하면 사용자가 **"이런 일이 생기면 이렇게 대응한다"는 플레이북**을 만들 수 있어요.

### FE: 너무 많은 JSON 파일은 관리가 어려울 수 있어요

**반론**: JSON 파일이 4-5개로 늘어나면 초보 사용자가 혼란스러워할 수 있습니다.

**대안 제안:**
1. **MVP에서는 `data.json` 하나로** — nodes, links, categories를 한 파일에
2. **향후 확장 시** events.json, scenarios.json 분리
3. **관리 UI를 제공** — JSON 직접 편집 대신 웹에서 폼으로 추가/수정

### DA: 합리적인 절충안이네요

동의합니다. MVP에서는:

```
data/
  data.json       — 노드 + 링크 + 카테고리 (통합)
```

Phase 2에서:
```
data/
  data.json       — 기본 데이터
  events.json     — 이벤트 기록
  scenarios.json  — 시나리오
  user-notes.json — 사용자 메모 (로컬스토리지 백업)
```

### PM: 노드 위치(레이아웃) 문제는 어떻게 할까요?

현재 x, y 좌표가 하드코딩인데, 노드가 늘어나면 수동 배치가 불가능합니다.

### FE: 두 가지 옵션이 있습니다

1. **Force-directed layout (자동 배치)**: D3.js force simulation으로 자동 배치. 노드가 많아져도 자동 적응. 단, 매번 약간 다른 레이아웃이 나올 수 있음.
2. **카테고리 기반 고정 레이아웃**: 카테고리별로 영역을 정해놓고 그 안에서 배치. 일관성 있지만 확장 시 조정 필요.

**추천: 하이브리드**
- 기본은 force-directed (자동)
- 사용자가 드래그해서 위치 조정한 건 localStorage에 저장
- 리셋 버튼으로 자동 레이아웃으로 복귀 가능

### ECO: 카테고리 구조도 정리가 필요합니다

현재 10개 카테고리인데, 체계가 좀 불분명해요. 제안:

**대분류:**
- 정책: 금리, 통화정책
- 시장가격: 채권, 주식, 원자재, 환율
- 경제지표: 물가, 고용, 성장
- 수급/심리: 외국인매매, VIX, 투자심리
- 외부충격: 전쟁, 자연재해, 팬데믹

이 분류는 향후 노드가 늘어날 때 정리의 기준이 됩니다.

## 합의된 데이터 구조

### Phase 1 (MVP): `data/data.json` 단일 파일

```json
{
  "version": "1.0",
  "lastUpdated": "2026-04-03",
  "categories": {
    "정책": { "color": "#60a5fa", "group": "정책" },
    "채권": { "color": "#a78bfa", "group": "시장가격" },
    ...
  },
  "nodes": [ ... ],
  "links": [ ... ]
}
```

### Phase 2: 이벤트/시나리오 분리
- `data/events.json`
- `data/scenarios.json`

### Phase 3: 사용자 커스텀 데이터
- localStorage 기반 사용자 메모/노드 위치/추가 관계

### 핵심 설계 원칙
1. **필수 필드는 최소화** — 노드는 4개 필드(id, label, category, description)만 필수
2. **선택 필드로 풍부화** — 강도, 시간 프레임, 조건 등은 있으면 좋지만 없어도 동작
3. **데이터와 뷰 분리** — 레이아웃 좌표는 데이터에 포함하지 않음 (force-directed 사용)
4. **하위 호환성** — 새 필드 추가해도 기존 데이터 깨지지 않는 구조
