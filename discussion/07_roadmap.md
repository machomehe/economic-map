### DA: Phase 1의 파일 구조를 정리하면

```
economic-map/
├── index.html          — 메인 (최소한의 HTML)
├── css/
│   └── style.css       — 스타일
├── js/
│   ├── app.js          — 초기화, 이벤트 바인딩
│   ├── graph.js        — D3 force 렌더링
│   ├── simulation.js   — 시뮬레이션 로직 (BFS, 부호 전파)
│   └── ui.js           — 팝업, 필터, 검색
├── data/
│   └── data.json       — 노드 + 링크 + 카테고리
└── discussion/         — 이 토론 기록
```

빌드 도구 없이, 브라우저에서 바로 실행. ES modules (type="module") 사용 가능.

### PM: 단, ES modules는 로컬 파일 시스템에서 안 됩니다

`file://` 프로토콜에서 ES modules는 CORS 에러가 납니다. GitHub Pages에서는 되지만 로컬 테스트가 불편해요.

**대안:**
1. ES modules 사용 + 로컬은 `python -m http.server`로 테스트 (코딩 초보한테 설명 필요)
2. 전통적 `<script>` 태그 순서 로딩 (빌드 없이 확실히 동작)
3. 단일 HTML 유지하되 데이터만 외부 JSON (절충안)

### FE: 옵션 3이 현실적입니다

```html
<script src="https://d3js.org/d3.v7.min.js"></script>
<script>
// data.json을 fetch로 로드
fetch('data/data.json')
  .then(r => r.json())
  .then(data => init(data));
</script>
```

이러면 HTML은 하나지만, **데이터는 외부 JSON**. JS 파일 분리는 Phase 2에서 해도 됩니다. MVP에서는 실용주의가 중요해요.

### DA: 동의합니다. 과도한 구조화보다 동작하는 코드가 먼저.

### PM: 각 팀원의 최종 의견을 들어볼까요?

### ECO: 데이터의 질이 핵심입니다
Phase 1에서 data.json의 노드/링크 데이터를 충실하게 작성하는 게 기술 구현보다 중요합니다. 특히 각 관계의 description, conditions, exceptions 필드를 꼼꼼히 채워야 학습 도구로서 가치가 있어요. 이 부분은 제가 감수할 수 있는 영역이니 데이터 초안을 제공하겠습니다.

### FE: 기술 스택 최소화가 생존 전략입니다
빌드 도구 없이, CDN + 정적 파일만으로, GitHub Pages에서 바로 동작. 이 제약을 지키는 한 사용자가 유지보수할 수 있습니다. D3.js 하나만 외부 의존성으로, 나머지는 순수 JS/CSS.

### DA: 데이터 파일의 스키마 문서화가 필요합니다
data.json의 각 필드가 뭘 의미하는지, 어떤 값이 가능한지 JSON Schema나 간단한 주석으로 문서화해야 합니다. 나중에 AI가 데이터를 자동 생성할 때도 이 스키마를 참조하게 됩니다.

### PM: 마지막으로, **첫 번째 작업의 우선순위**

Phase 1의 Sprint 1-1이 가장 먼저이고, 그중에서도:
1. `data/data.json` 생성 (데이터 이관 + 필드 확장)
2. D3.js force-directed layout 구현
3. 기본 인터랙션 (탭 → 하이라이트)

이 세 가지가 완성되면 나머지는 점진적으로 붙일 수 있습니다.

## 합의된 로드맵 요약

| Phase | 기간 | 핵심 산출물 | 우선순위 |
|-------|------|------------|---------|

### 기술 스택
- HTML + CSS + Vanilla JS (빌드 도구 없음)
- D3.js v7 (CDN)
- GitHub Pages (배포)
- localStorage (사용자 데이터)
- Claude API (Phase 2, AI 분석)
